# Acceptable Use Policy

**Version**: 1.0
**Effective**: 2026-05-10
**Owner**: Founder
**Applies to**: All workforce members (currently founder only) and contractors with access to referme.help systems

## Purpose

Define permitted and prohibited uses of company systems and data. Maps to HIPAA §164.308(a)(4) (Information Access Management) and §164.530(b) (Workforce Training).

## Acceptable use

You may use company systems for:

- Performing work duties as defined in your role.
- Necessary administration (account management, support).
- Reasonable personal use that doesn't interfere with work, consume excessive resources, or expose the company to risk (this clause is here for when there's a "you" other than the founder).

## Prohibited use

You may NOT:

- Access PHI or other production data without a documented work need ("least privilege" — if you don't need it for a task, don't open it).
- Share credentials with anyone (yourself across devices is fine via 1Password; humans across humans is not).
- Disable, bypass, or modify security controls (HSTS, audit logging, scope checks, rate limits, CI gates).
- Install unapproved software on systems with PHI access — including on developer laptops if the laptop is used to access prod.
- Use personal cloud storage (Dropbox, Google Drive, iCloud) for company data.
- Take screenshots of real customer data for any purpose.
- Bypass code review for non-emergency changes.
- Use AI tools (Copilot, Cursor, ChatGPT, Claude) on production data unless the AI vendor has a BAA on file.
- Forward company email containing PHI to personal accounts.
- Discuss customer-identifying information in social media, public communities, conference talks, etc.

## Device requirements

Any device used to access production:

- Full-disk encryption (FileVault on macOS, BitLocker on Windows, LUKS on Linux).
- Auto-lock ≤ 5 minutes idle.
- Up-to-date OS + browser (within 30 days of latest stable).
- Antivirus / EDR (macOS built-in XProtect acceptable; Linux: ClamAV or equivalent).
- Local firewall on.
- Bluetooth file sharing off.

## Authentication

- Strong passwords (≥16 chars, generated by 1Password / equivalent).
- 2FA on every account that supports it.
- No reuse of passwords across systems.

## Network

- Avoid public Wi-Fi for production access; if necessary, use a personal hotspot or a reputable VPN.
- Don't admin production from a coffee shop without VPN.

## Data handling

See `data-classification.md` for tier-by-tier rules. Highlights:

- Never put PHI in URLs, session cookies, logs, error responses, transactional email bodies, third-party analytics, or unapproved support systems.
- Use synthetic data in screenshots, demos, and test fixtures.
- Don't commit credentials to repos (gitleaks gate).

## Reporting

If you suspect a security incident, lost device, compromised credential, or policy violation:

- Report immediately to the founder (currently the same person — write it down anyway in `~/Documents/robworks/incidents/`).
- Don't try to "handle it yourself" if there's any chance PHI is involved.
- Don't delete evidence.

## Acknowledgement

When the workforce is > 1, this policy will be acknowledged in writing by every workforce member at hire and annually thereafter. Acknowledgements stored in `docs/compliance/acknowledgements/`.

For now, the founder acknowledges by virtue of authoring it.

## Enforcement

Violations are addressed proportionally:

- Minor / inadvertent: coaching + corrective action.
- Major / negligent: formal warning + remediation plan.
- Willful or repeated: termination of access + relationship.

For contractors, the same scale applies via the contractor agreement.
