## Build & Test
- `pip install -e .` — install CLI
- `pip install -e ".[web]"` — install CLI + web UI
- `pip install -e ".[dev]"` — install with test deps
- `pytest` — run all tests
- `ruff check .` — run lint checks
- `docstats web` — start web UI at http://127.0.0.1:8000

## Architecture
- `src/docstats/` layout — one module per concern, each under 300 lines; web routes split into `routes/` subpackage
- Core modules (client, models, storage, cache, normalize, scoring) have zero dependency on Rich/Typer/FastAPI
- Web layer imports core modules directly — no wrappers or adapters
- Dual-backend storage: Supabase Postgres in production (when `SUPABASE_URL` + `SUPABASE_SERVICE_KEY` are set), SQLite locally (default)
- SQLite DB at `~/.local/share/docstats/docstats.db` (local dev/CLI only)
- Supabase tables use `docstats_` prefix (e.g. `docstats_users`) to coexist with robworks-software in shared project (ref: `uhnymifvdauzlmaogjfj`)
- `get_storage()` returns `PostgresStorage` or `Storage` (SQLite) — both inherit from `StorageBase` ABC in `storage_base.py`; consumers should type-hint `StorageBase`
- `storage_base.py` contains `StorageBase` ABC + shared helpers (`normalize_email`, `fuzzy_score`); `storage.py` contains `Storage(StorageBase)` (SQLite) and `get_storage()`; `pg_storage.py` contains `PostgresStorage(StorageBase)` (Supabase)
- `src/docstats/domain/` is the referral-platform domain layer. Keep it free of FastAPI dependencies where possible; FastAPI deps live in `routes/`
- Audit log: `domain/audit.py` exposes `AuditEvent` model + `record(storage, *, action, request=None, ...)` helper that swallows errors. Action vocabulary is `{entity}.{verb}` (e.g. `user.login`, `provider.save`). Storage methods `record_audit_event` / `list_audit_events`; rows are append-only by contract. Postgres table `docstats_audit_events`; SQLite `audit_events`. FKs use `ON DELETE SET NULL` so audit rows survive user deletion
- Dual-mode scope: `docstats/scope.py` defines the `Scope` frozen dataclass + `scope_sql_clause()` helper + `ScopeRequired` exception. `routes._common.get_scope` is the FastAPI dep; silently falls back to solo mode if `active_org_id` is stale. Every Patient/Referral row from Phase 1+ carries exactly one of `scope_user_id` or `scope_organization_id` — enforced by CHECK constraint on the DB and scope filter on every read/write. SQLite callers use `scope_sql_clause(scope)` for `(fragment, params)`; Postgres callers use `PostgresStorage._apply_scope(query, scope)` to wrap a supabase-py chain with `.eq(...).is_(..., None)`
- Patients (Phase 1.A): `domain/patients.py` `Patient` model; storage methods take `Scope` first-arg and scope-filter every query. Anonymous scope raises `ScopeRequired`. MRN unique within an org (partial index, live rows only). `update_patient` treats `None` kwargs as "leave unchanged"
- Referrals (Phase 1.B): `domain/referrals.py` — `Referral`/`ReferralEvent` models + enum constants + `STATUS_TRANSITIONS` state-machine map + `transition_allowed`/`require_transition` helpers. Storage: `create_referral` / `get_referral` / `list_referrals` (filter by patient/status/assignee) / `update_referral` / `set_referral_status` (dumb — validate transition first via `require_transition`) / `soft_delete_referral` / `clear_referral_field(scope, id, field)` (explicit clear for nullable fields — companion to `None`-means-skip on `update_referral`; only clearable names: `assigned_to_user_id`, `authorization_number`, `payer_plan_id`, `external_reference_id`, `diagnosis_primary_icd`, `diagnosis_primary_text`) / `record_referral_event` / `list_referral_events`. Cross-scope `patient_id` AND `payer_plan_id` on create/update raise `ValueError` — both are scope-owned FKs. `created` event auto-seeded on insert; SQLite wraps row+event in a single `with self._conn:` transaction, Postgres uses a compensating soft-delete on event-insert failure so a partial write never leaves a referral without a `created` event. Events scope-transitively through referral — no scope cols on event row. Patient FK is `ON DELETE RESTRICT` (referrals are evidence)
- Referral clinical sub-tables (Phase 1.C): `referral_diagnoses`, `referral_medications`, `referral_allergies`, `referral_attachments` — all hang off referrals via CASCADE, no scope columns (scope flows via parent referral's `get_referral` gate). 16 storage methods (add/list/update/delete × 4). Every row has `source` from `SOURCE_VALUES`; attachments also have `kind` from `ATTACHMENT_KIND_VALUES`. Hard-delete not soft — `referral_events` captures edits. One primary diagnosis per referral enforced by partial unique index. Headline sync: add/update/delete on `referral_diagnoses` with `is_primary=True` (or delete of the current primary row, or edit of its code/desc) auto-syncs `referrals.diagnosis_primary_icd` / `diagnosis_primary_text` via `_sync_referral_primary_diagnosis` — sub-table is source of truth, the denormalized headline columns on `referrals` never drift
- Referral responses (Phase 1.D): `referral_responses` — closed-loop updates (appointment_date, consult_completed, recommendations_text, received_via, recorded_by_user_id). Multiple per referral; `RECEIVED_VIA_VALUES` = fax/portal/email/phone/manual/api. Storage: `record_`/`list_`/`update_`/`delete_referral_response`. `consult_completed=True` signals the route layer to move referral → status=completed; storage doesn't cascade the status change itself
- Reference data (Phase 1.E): `domain/reference.py` — `InsurancePlan` (scope-owned), `SpecialtyRule` + `PayerRule` (nullable `organization_id`; `NULL` = platform default, int = org override). Partial unique indices per (global) and per (org, key). `list_*_rules(organization_id=X, include_globals=True)` returns both globals AND X's overrides as separate rows, ordered `(code, organization_id NULLS FIRST, id)` so globals precede overrides for the same code — the rules engine (Phase 3) owns the "org override wins" merge; pass `include_globals=False` for org-only. `update_*_rule` takes `bump_version: bool = True` (canonical-restore callers pass False so caches stay valid) and `overwrite: bool = False` (True writes every kwarg literally, including `None`, so seed re-runs can reset a field back to `NULL` over an admin-set value). JSONB in Postgres → TEXT-JSON in SQLite via `_parse_jsonb`. Rules hard-delete; admin audit covers it
- Export pipeline (Phase 5.A–5.E): `src/docstats/exports/pdf.py` ships five artifact constants + five `render_*` fns returning PDF bytes (`render_referral_summary`, `render_scheduling_summary`, `render_patient_summary`, `render_attachments_checklist`, `render_missing_info`). Shared helpers `_render_pdf` + `_base_context` keep templates consistent. FastAPI-free — `routes/exports.py` owns scope/audit/HTTP and dispatches via `_ARTIFACT_RENDERERS` (maps artifact string → (fetcher, renderer, filename_stem)); adding a new artifact is one tuple append. Missing-info fetcher calls `rules_based_completeness(storage, scope, referral)` so the PDF reflects Phase 3 rules. `GET /referrals/{id}/export.pdf?artifact=<name>` is PHI-consent gated + scope-enforced; WeasyPrint render wrapped in `loop.run_in_executor`. Audit emits `referral.export` (metadata: artifact/format/bytes); referral event `event_type="exported"`. Filename server-controlled (`referral-{id}-{stem}.pdf`, no PHI). Unknown artifact → 400; missing referral → 404; scope-gated missing patient → 409; oversized artifact Query → 422. Templates at `templates/exports/*.html`; shared print CSS at `static/print.css` adds 5.B classes (`.hero-card`, `.patient-*`, `.alert-card`). Detail page exposes individual artifacts via `<details class="export-menu">` + "Export packet…" button to the preview page at `GET /referrals/{id}/export`. Packet is a pypdf concatenator; `?include=a,b,c` controls ordering (default: fax_cover,summary,attachments). Unknown/nested artifacts in include → 400. Needs `pypdf>=6.0`. Phase 5.D adds `src/docstats/exports/fhir.py` with `build_referral_bundle(...)->dict` and `GET /referrals/{id}/export.json` returning a FHIR R4-shaped Bundle (Patient + ServiceRequest + Practitioner + Organization + Condition + MedicationStatement[] + AllergyIntolerance[] + DocumentReference[]). "FHIR-ish" not strict-validator-passing. `_STATUS_MAP` pinned by a test so new STATUS_VALUES don't silently drop to `unknown`. Phase 5.E adds `src/docstats/exports/csv_export.py` with `CSV_FIELDNAMES` + `referral_to_csv_row(referral, patient)->dict`, `GET /referrals/export.csv` (same filters as workspace; cap 2000 rows) and `POST /referrals/batch-export.pdf` (`referral_ids=1,2,3` comma-separated Form field, cap 50, default artifact=summary, per-ID errors land in `X-Export-Skipped` header + audit). Literal routes registered BEFORE `/{referral_id}/...` so FastAPI doesn't try to coerce "export.csv" to int. Phase 5 is done
- WeasyPrint system deps: Railway build needs Pango/Cairo shared libs via `railpack.json` at repo root with BOTH `buildAptPackages` and `deploy.aptPackages` set to `[libpango-1.0-0, libpangoft2-1.0-0, libharfbuzz0b, libcairo2]` — runtime needs them at container start, not just pip-build. Coexists with `railway.toml` (which still owns build/start commands). Local macOS: `brew install pango cairo`. Tests gate WeasyPrint import with `pytest.importorskip("weasyprint", ...)` so lint-only shards pass
- Admin console (Phase 6.A): `routes/admin.py` + `require_admin_scope` dep (403 for solo users and sub-admin org roles; AuthRequiredException → /auth/login for anonymous). `GET /admin` renders `templates/admin/overview.html` (extends `admin/_admin_layout.html` — sticky sidebar with nav to 6.B–6.F surfaces marked "Coming soon"). Stats: active member count, specialty/payer rule globals + org overrides, last 10 audit events for `scope_organization_id`. Nav visibility: `base.html` shows the top-level Admin link only when `user.is_org_admin`. `get_current_user` in `auth.py` computes that flag by looking up the active-org membership — one extra DB hit per authenticated request when active_org_id is set; lookup failures log + degrade to False. CSS under `/* ── Admin console (Phase 6.A) ── */` in `static/style.css`. Router include: `admin_router` after `api_router`, before `patients_router`. Single choke point for role enforcement — do NOT re-check roles inside handlers.
- Specialty-rules editor (Phase 6.B): adds `GET /admin/specialty-rules` (list), `GET/POST /admin/specialty-rules/{code}` (edit form + save), `POST /admin/specialty-rules/{code}/revert` (hard-delete override; idempotent no-op emits no audit). Edit form seeds from existing override if present, else from global — saving creates a faithful copy. Unknown `required_field` names filtered at route boundary against `REQUIRED_FIELD_CHECKS` (renamed from private `_REQUIRED_FIELD_CHECKS` in `domain/rules.py`; private alias retained for one release). `recommended_attachments.kinds` preserved verbatim from the seeding source so seed data isn't dropped by the UI (which only exposes `labels`). Audit actions: `admin.specialty_rule.{create,update,revert}_override`. Templates: `admin/specialty_rules_list.html` + `admin/specialty_rule_edit.html`. Tests gotcha: httpx `data={"key": ["v1","v2"]}` dict form reaches the handler; `data=[(...)]` tuples don't.
- CSV imports (Phase 1.F): `domain/imports.py` — `CsvImport` (scope-owned) + `CsvImportRow` (scope-transitive via parent). `IMPORT_STATUS_VALUES` = uploaded/mapped/validated/committed/failed; `IMPORT_ROW_STATUS_VALUES` = pending/valid/error/committed/skipped. State-machine maps + `require_*_transition` helpers; storage stays dumb. `(import_id, row_index)` UNIQUE guards parsing double-inserts. `referral_id` FK is SET NULL (preserves import provenance). `update_csv_import_row(referral_id=...)` raises `ValueError` on cross-scope referral IDs — mirrors the patient/payer-plan FK guards on referral mutations. Both tables hard-delete — ephemeral staging data
- Seed defaults (Phase 1.G): `domain/seed.py` — `SPECIALTY_DEFAULTS` (12) + `PAYER_DEFAULTS` (8) + `seed_platform_defaults(storage, *, overwrite=False)` idempotent seeder. Default path pre-fetches existing globals into a code/key map (O(N), not O(N²)) and skips known codes. `overwrite=True` calls `update_*_rule(..., bump_version=False, overwrite=True)` so canonical restores don't invalidate rule-engine caches and can reset admin-set values back to `None`. Not auto-run at deploy — production seeding happens when Phase 3 rules engine lands. `tests/test_scope_isolation_e2e.py` proves cross-tenant isolation across the full Phase 1 entity graph; `tests/test_phase1_review_fixes.py` pins the retrospective-review bugfixes
- Postgres scope helpers: `PostgresStorage._require_scoped(scope)` is the single choke-point for the "non-anonymous scope required" rule on insert paths (replaces inline `if scope.is_anonymous: raise` scattered across `create_patient` / `create_referral` / `create_insurance_plan` / `create_csv_import`). `_apply_scope` on a query chain calls it internally so SELECT/UPDATE/DELETE paths are covered the same way
- Organizations + memberships: `domain/orgs.py` holds `Organization`, `Membership`, `ROLES` ladder (read_only → staff → clinician → coordinator → admin → owner), `has_role_at_least()`. Slugs use a partial unique index (live rows only). `memberships.(organization_id, user_id)` is a plain UNIQUE — `create_membership` uses `ON CONFLICT DO UPDATE` (SQLite) / `.upsert(on_conflict="organization_id,user_id")` (Postgres) to reactivate a soft-deleted row
- Server-side sessions: `domain/sessions.py` `Session` model; storage methods `create_session` (URL-safe 32-byte token id), `get_session`, `touch_session`, `revoke_session`, `list_sessions_for_user`, `purge_expired_sessions`. ## NPPES API
- Endpoint: `https://npiregistry.cms.hhs.gov/api/?version=2.1`
- Cannot combine NPI-1 fields (first_name, last_name) with NPI-2 fields (organization_name)
- No middle_name search param — use for post-retrieval ranking only
- Returns names UPPERCASE, phones unformatted, postal codes raw (9 digits)
- API has no auth, max 1200 results per query
- Name searches match against `other_names` (former/maiden names) too — results sorted alphabetically by current name, so common names (Smith, Patel) return mostly former-name matches first; prefix-filter results by current name when relevance matters
- `taxonomies.py` has 883 NUCC taxonomy descriptions for client-side specialty autocomplete

## Gotchas
- All `/provider/{npi}/...`, `/onboarding/select-pcp/{npi}`, `/profile/pcp/{npi}` routes must use `npi: str = Depends(require_valid_npi)` (from `docstats.validators`) — never raw `npi: str`; keeps malformed input at the boundary (422) and closes header-injection in export filenames. Same for new Form/Query fields: add `max_length=` per the caps in `docs/security-audit.md`
- SQLite connections need `check_same_thread=False` for FastAPI/uvicorn
- SQLite WAL mode (`PRAGMA journal_mode=WAL`) required for concurrent CLI+web access
- SQLite FK enforcement is off by default — `PRAGMA foreign_keys = ON` must be set per-connection (after WAL pragma in `Storage.__init__`)
- `INSERT OR REPLACE` triggers FK cascade deletes on the replaced row — use `INSERT ... ON CONFLICT(col) DO UPDATE SET ...` upsert to preserve child rows
- Multi-step SQLite writes that must land together (row + seed event, sub-table + denormalized headline, etc.) MUST be wrapped in `with self._conn:` — the context manager commits on normal exit and rolls back on any raised exception. Storage helpers that participate in such a transaction (e.g. `_sync_referral_primary_diagnosis`) do NOT call `self._conn.commit()` themselves; committing is the caller's job. Mixing the two styles (`with self._conn:` around an inner helper that also commits) flushes the transaction early — usually fine, but it defeats rollback if a later statement in the `with` block raises. For Postgres, there's no single-statement transaction over the REST API — use explicit compensation (see `create_referral` soft-delete fallback) and swallow secondary compensation errors so the original exception propagates unchanged
- Same-second SQLite timestamps make `ORDER BY created_at DESC` non-deterministic in tests — always include `id DESC` as a tiebreaker
- Route ordering via `app.include_router()` in `web.py` matters: `saved_router` must be included before `providers_router` so `/saved/export/csv` matches before `/provider/{npi}`; within each router, specific routes (e.g., `/{npi}/export`) must be declared before parameterized routes
- htmx `HX-Target` header includes the `#` prefix from CSS selectors
- Pydantic `basic` field is a dict (not typed) because NPI-1 and NPI-2 have incompatible schemas — use `parsed_basic()` method
- Starlette 0.50+ changed `TemplateResponse` signature — use `render()` from `routes/_common.py` instead of calling `templates.TemplateResponse()` directly
- When adding new storage methods, add the abstract method to `StorageBase` first — both `Storage` and `PostgresStorage` inherit from it and must implement all abstract methods
- `normalize_email()` and `fuzzy_score()` live in `storage_base.py` — use these instead of inline `email.strip().lower()` or reimplementing fuzzy matching
- All HTTP clients (NPPES, OIG, CMS, Open Payments) use `request_with_retry()` from `http_retry.py` for exponential backoff retry — NPPES translates `_NPPESRetryExhausted` into user-facing `NPPESError` in `client._friendly_transport_error`
- Retry tests must patch `client._http.request` (not `.get`/`.post`) and `docstats.http_retry.time.sleep` — retry logic is in `http_retry.py`, not in each client module
- HTTP timeout/retry knobs come from env vars (read at call time, not import): `DOCSTATS_HTTP_TIMEOUT` (default 30.0s) for httpx client timeout; `DOCSTATS_HTTP_MAX_RETRIES` (default 3) for retries on 429/5xx/timeout; `DOCSTATS_HTTP_CONCURRENCY` (default 5) for the semaphore size returned by `docstats.concurrency.async_limiter()` for batch callers. New httpx clients should read `get_default_timeout()` at `__init__`; new batch paths should consume `async_limiter()` instead of unbounded `asyncio.gather`
- `NPPESClient.async_lookup_many(npis, *, limiter=None)` is the batch lookup seam — results in input order, capped by the passed semaphore or `async_limiter()`. Pass a shared `asyncio.Semaphore` when batching from a long-running process so the cap is shared across calls
- `request_with_retry()` honors integer-seconds `Retry-After` headers on retryable statuses (≥ 0.5s); HTTP-date form is not supported
- `http_retry.py` `backoff_base` is an *exponent base*, not a constant multiplier — delays are `backoff_base ** attempt`. Default `2.0` → `1s, 2s, 4s`; setting `backoff_base=1.0` silently degrades to `1s, 1s, 1s` (`1**n == 1`). If you change it, update `test_exponential_backoff_delays` and the docstring together
- To smoke-test NPPES retry error messages without a real network failure: `DOCSTATS_HTTP_TIMEOUT=0.001 DOCSTATS_HTTP_MAX_RETRIES=0 docstats web` — forces the "took too long" `NPPESError` path instantly
- New routes go in the appropriate `routes/*.py` file, not in `web.py` — only app-level middleware, exception handlers, and router includes belong in `web.py`
- Don't use `pip freeze` on this machine for generating `requirements.txt` — global env has 500+ unrelated packages
- `httpx.TimeoutException` is a subclass of `RequestError` — catch it first or it's dead code
- Templates must guard against `None` models — routes pass `result=None` on `NPPESError`
- `scoring.py` result ranking is integrated into both the Web UI (via `routes/search.py`) and the CLI (via `services.py` `search_providers`)
- `querySelectorAll('input')` does not match `<select>` elements — when clearing a form section, reset selects explicitly (e.g. `el.value = ''`) in addition to iterating inputs
- Use `clearSuggestions(id)` to clear suggestion lists in `index.html` — do not set `.innerHTML = ''` directly
- To trigger `initAutocomplete`'s `activeIdx` reset for a non-htmx list, dispatch `new Event('htmx:afterSwap')` on the list element after populating it
- JS that references elements inside `{% if ... %}` blocks must null-guard or be inside the same conditional — the element won't exist when the condition is false
- Mapbox Geocoding: for `postcode`-type features, the ZIP is in `f.text` not `f.context` — add `if (!zip && place_type is postcode) zip = f.text` alongside the `place`-type city fallback
- Mapbox tokens: `pk.` = public (safe for client-side JS), `sk.` = secret (server-side only)
- To safely inject a Python template variable into JS, use `{{ var | tojson }}` — handles escaping
- CSS `:has()` requires `@supports selector(:has(*))` guard — without it, hiding inputs styled only via `:has(input:checked)` leaves no visual feedback on Firefox <121; wrap both the `display:none` and the `:has()` rule together inside the `@supports` block
- CSS utility classes used in multiple templates must be defined standalone (e.g. `.back-link { ... }`) not only as descendant selectors (e.g. `.action-bar .back-link`) — descendants work only inside that specific parent; silently no-ops elsewhere
- `hx-swap="outerHTML"` on a button inside a named `<div id="...">` destroys that container ID — subsequent htmx clicks target a missing element; use `hx-swap="innerHTML"` on the container instead
- `_save_button.html` uses `btn_target` variable (container ID without `#`) so it works from multiple call sites; routes pass it via `request.headers.get("hx-target", "#save-btn").lstrip("#")`
- History re-run links navigate to `/?query=...`; `index.html` has a `DOMContentLoaded` handler that reads `?query=` from the URL and auto-triggers `htmx.trigger(form, 'submit')` — required for re-run to land on results
- In the smart-search path (`query` param), `rank_results()` must receive a `SearchQuery` built from the winning `interp` dict (`first_name`, `last_name`, `organization_name`, `taxonomy_description`), not the empty structured-form fields
- `require_user` dependency raises `AuthRequiredException`; the exception handler in `web.py` returns `HX-Redirect` header (200) for HTMX requests, and `303` redirect for normal requests — HTMX doesn't follow 3xx redirects correctly
- Anonymous saves hit `POST /provider/{npi}/save` which returns `_auth_gate.html` (not a redirect) so HTMX can swap it inline; never use `require_user` on this route
- `SESSION_SECRET_KEY` not set → random key generated at startup (dev-only fallback); sessions won't survive server restarts without it set in env
- `saved_providers` migration: `_migrate_saved_providers()` checks `PRAGMA table_info` for `user_id`; if absent, drops and recreates with composite PK — existing data is lost (acceptable on Railway due to ephemeral filesystem)
- All full-page routes must pass `user=current_user` in template context for `base.html` nav to render correctly
- Test auth override: `app.dependency_overrides[get_current_user] = lambda: fake_user_dict` — `require_user` inherits this automatically since it depends on `get_current_user`
- Adding a column to `saved_providers` requires updates in: (1) `storage_base.py` abstract method (if new public API), (2) `storage.py` migration + `save_provider` INSERT + `_row_to_provider`, (3) `pg_storage.py` `_row_to_provider` + `save_provider` upsert dict + preserve-on-conflict fetch, (4) `models.py` `SavedProvider` field + `export_fields()`, (5) `routes/saved.py` `_CSV_FIELDNAMES` + all relevant route template contexts
- `saved.html` includes `_appt_address.html` via `{% set npi %}{% set appt_address %}{% set appt_suite %}{% set appt_phone %}{% set appt_fax %}{% set is_televisit %}{% include %}` — any new context variable for that partial must be added to the `{% set %}` chain AND to all `render("_appt_address.html", {...})` calls in `routes/providers.py`
- `pg_storage.py` has no auto-migration — new columns must be added to Supabase manually via Management API SQL endpoint before deploying code that references them. Migration SQL files live in `docs/migrations/NNN_name.sql`; apply with `curl -sS -X POST "https://api.supabase.com/v1/projects/uhnymifvdauzlmaogjfj/database/query" -H "Authorization: Bearer $SUPABASE_ACCESS_TOKEN" -H "Content-Type: application/json" --data "$(jq -Rs '{query: .}' docs/migrations/NNN.sql)"`. Python `urllib.request` trips Cloudflare bot protection (HTTP 403, error 1010) — use curl, not urllib
- Postgres column-level `UNIQUE` is unconditional — if you want "live rows only" uniqueness, use a partial unique index (`CREATE UNIQUE INDEX ... WHERE deleted_at IS NULL`) AND drop the column-level `UNIQUE`. Both together silently block reuse; the column constraint wins (migration 005 fixes a shipped-then-caught instance)
- supabase-py composite-key upsert syntax: `.upsert(row, on_conflict="col1,col2")` — comma-delimited string, not a list
- supabase-py `.delete()` does NOT return deleted rows — `result.data` is always `[]`. To get a count, select ids first then delete (pattern used by `purge_expired_sessions`)
- supabase-py pagination: `.range(start, end)` is INCLUSIVE + zero-indexed; for offset/limit use `.range(offset, offset + limit - 1)` (used by `list_patients`)
- Date-only columns (`date_of_birth` etc.): Postgres `DATE` + SQLite `TEXT`, both return ISO `YYYY-MM-DD` strings via their clients; Pydantic models use `str | None`
- Every `_row_to_*` helper in `storage.py` must route timestamp columns through `_parse_sqlite_utc` (not raw `datetime.fromisoformat`) so SQLite datetimes are tz-aware UTC like Supabase output; mixing aware/naive raises `TypeError` on comparisons with `datetime.now(tz=timezone.utc)`
- When adding a storage method that inserts a row, prefer `ON DELETE SET NULL` over `ON DELETE CASCADE` on audit / log / immutable-history tables — cascading deletes silently destroy evidence. `CASCADE` is fine for purely derived data like `saved_providers` (per-user rolodex)
- Any new keys added to `SavedProvider.export_fields()` must also be added to `_CSV_FIELDNAMES` in `routes/saved.py` — `DictWriter` raises `ValueError` on extra keys by default
- `passlib[bcrypt]` is incompatible with `bcrypt>=4.0.0` — pin `bcrypt>=3.2.0,<4.0.0` in both `requirements.txt` and `pyproject.toml` web extras; bcrypt 4.x raises `ValueError` on passwords >72 bytes instead of silently truncating
- `python-multipart` must be explicit in `requirements.txt` — FastAPI requires it for any form POST route; Railpack won't install it as a transitive dep
- CSS input styles in `base.html` must enumerate `input[type="email"]` and `input[type="password"]` explicitly — they don't inherit from `input[type="text"]` rules
- Railway build environment uses Python 3.13 — passlib prints a `crypt` deprecation warning on startup; harmless but expected
- Onboarding gate checks `terms_accepted_at` (DB column), not `pcp_npi` — GitHub OAuth bypass must also check `terms_accepted_at` to avoid skipping terms acceptance
- `_onboarding_step()` accepts `pcp_skipped` kwarg from session — if PCP was skipped, step 3 is bypassed on resume so user lands on step 4 (terms) instead of looping back to PCP
- `date_of_birth` must be validated server-side (`date.fromisoformat()` + future-date check) — the HTML `max` attribute is client-side only
